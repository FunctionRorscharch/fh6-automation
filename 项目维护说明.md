# 项目维护说明

本文档用于说明当前项目的文件职责、常见维护入口，以及后续改动时的建议方式。

## 1. 项目总体结构

当前项目可以理解为 6 层：

1. 总控层  
   负责程序启动、UI 初始化、配置管理、日志、诊断、自适应校准、三条业务流调度。

2. 业务流程层  
   负责跑图、买车、超级抽奖三条主流程。

3. 图像识别层  
   负责模板匹配、灰度识别、透明图识别、多尺度识别、特殊目标识别。

4. 公共动作层  
   负责多个流程共享的小动作与等待逻辑。

5. 资源与参数层  
   负责图片目录、缓存目录、配置路径、识别参数中心。

6. UI 布局层  
   负责控件布局、输入框、开关、日志区域与状态显示。

## 4. 识别参数说明

项目里最常见的几个识别参数：

### `threshold`

匹配阈值。

含义：
- 越高越严格
- 越低越容易命中，但也更容易误判


### `timeout`

等待上限，不是固定延时。

含义：
- 在这段时间内持续尝试识别
- 一旦中途识别成功，会立刻返回
- 超时还没识别到，才算失败


### `interval`

两次识别尝试之间的间隔。

含义：
- 越小，检测更频繁，响应更快
- 越大，占用更低，但反应稍慢


### `fast_mode`

快速缩放匹配模式。

含义：
- `True`：尝试较少的缩放比例，速度更快
- `False`：尝试更多缩放比例，兼容性更高，但更慢

维护建议：
- 稳定场景用 `True`
- 易漏识别场景用 `False`


## 5. 现在推荐的维护顺序

当你发现某个地方有问题时，建议按这个顺序排查：

1. 先开诊断模式复现一次  
   先看诊断报告、运行日志、失败截图。

2. 先调参数，不先改逻辑  
   优先看 `recognition_config.py` 和 `config.json` 中的覆盖项。

3. 再看是否属于识图层问题  
   如果是命中不稳、缩放不准、图片不兼容，优先改 `image_matcher.py`。

4. 最后再改业务流程  
   如果确认识图没问题，只是流程顺序不合理，再改 `flow_race.py / flow_buy.py / flow_cj.py`。

## 7. 配置覆盖建议

以后如果只是想调少量识别参数，优先使用 `config.json` 覆盖：

```json
{
  "recognition_profiles": {
    "buy.consumablecar": {
      "threshold": 0.85
    },
    "race.eventlab": {
      "threshold": 0.68,
      "timeout": 6
    }
  }
}
```

这样做的好处：
- 不改源码
- 便于针对不同机器做快速调试
- 出问题时可直接回退


## 8. 后续维护建议

### 建议 1
继续把零散识别参数往 `recognition_config.py` 集中。

### 建议 2
业务流程文件只保留业务逻辑，不继续混入大量识图细节。

### 建议 3
识图失败时优先依赖诊断模式，而不是凭感觉猜。

### 建议 4
新增模板时尽量保持文件名稳定，减少联动修改成本。


## 9. 一个简单判断规则

以后你只要先问自己一句：

“我要改的是流程、识图、参数、资源，还是界面？”

通常就能直接定位文件：

- 流程 -> `flow_race.py / flow_buy.py / flow_cj.py`
- 识图 -> `image_matcher.py`
- 参数 -> `recognition_config.py`
- 资源 -> `app_resources.py`
- 界面 -> `ui_layout.py`
- 总控/配置/校准 -> `main.py`
